# 文档长度规范

## 1. 长度限制原则

### 1.1 核心原则

为了确保文档的可读性和维护性，所有 Skill 文档都应该遵循适当的长度限制。过长的文档会增加用户的阅读负担，同时也不利于维护和更新。

### 1.2 长度限制标准

| 文档类型 | 建议长度 | 最大长度 | 超出处理方法 |
|---------|---------|---------|-------------|
| 主题文档 | 50-100 行 | 150 行 | 拆分为多个文档 |
| 功能模块文档 | 100-200 行 | 250 行 | 拆分为多个文档 |
| 详细说明文档 | 150-250 行 | 300 行 | 拆分为多个文档 |

### 1.3 限制理由

- **可读性**：较短的文档更容易阅读和理解
- **维护性**：较短的文档更容易更新和维护
- **导航性**：较短的文档便于用户快速定位所需信息
- **一致性**：确保所有文档保持适当的长度

## 2. 文档拆分方法

### 2.1 拆分原则

当文档长度接近或超过限制时，应按照以下原则进行拆分：

- **逻辑相关性**：将相关的内容保持在同一文档中
- **功能独立性**：每个拆分后的文档应该具有独立的功能或主题
- **层级一致性**：拆分后的文档应该遵循相同的编号规范
- **引用完整性**：确保拆分后的文档之间建立适当的引用关系

### 2.2 拆分步骤

1. **识别拆分点**：
   - 寻找自然的逻辑断点，如章节末尾
   - 识别可以独立成篇的内容模块
   - 考虑用户的阅读习惯和信息需求

2. **创建新文档**：
   - 按照编号规范创建新的文档文件
   - 为新文档分配适当的编号
   - 遵循文档命名规范

3. **迁移内容**：
   - 将相关内容迁移到新文档中
   - 确保内容的完整性和连贯性
   - 更新文档中的引用和链接

4. **更新引用**：
   - 在相关文档中添加对新文档的引用
   - 确保用户可以方便地找到拆分后的内容

### 2.3 拆分示例

#### 原文档结构：
```
0001_usage_guide.md (300 行)
```

#### 拆分后结构：
```
0001_usage_guide.md (150 行)
0001_01_pdf_processing.md (100 行)
0001_02_llm_processing.md (100 行)
```

## 3. 文档精简技巧

### 3.1 内容精简

- **删除冗余信息**：移除重复或不必要的内容
- **合并相似内容**：将相似的内容合并为一个章节
- **使用表格**：将复杂的列表转换为表格
- **使用图表**：使用图表替代冗长的文字说明

### 3.2 结构优化

- **合理使用标题**：使用清晰的标题和小标题
- **使用列表**：将冗长的段落转换为列表
- **使用代码块**：将代码示例放入代码块中
- **使用引用**：对于详细信息，使用引用而非内联

### 3.3 语言精简

- **使用简洁的语言**：避免冗长的句子和复杂的词汇
- **使用主动语态**：使用主动语态使句子更简洁有力
- **避免重复**：避免重复表达相同的意思
- **保持一致性**：使用一致的术语和表达方式

## 4. 长度检查方法

### 4.1 手动检查

- 使用文本编辑器的行号功能检查文档长度
- 定期审查文档，确保长度符合要求
- 在更新文档时检查长度变化

### 4.2 自动检查

- 可以使用脚本工具自动检查文档长度
- 设置长度警告阈值，提前发现过长的文档
- 集成到 CI/CD 流程中，确保所有文档符合长度要求

## 5. 最佳实践

### 5.1 预防措施

- **规划先行**：在创建文档前规划内容和长度
- **逐步构建**：从核心内容开始，逐步扩展
- **定期审查**：定期检查文档长度，及时进行调整
- **用户反馈**：根据用户反馈调整文档长度和结构

### 5.2 特殊情况处理

- **技术深度**：对于技术深度较高的内容，可以适当增加长度
- **参考资料**：对于参考资料和附录，可以单独创建文档
- **示例代码**：对于较长的示例代码，可以考虑使用外部链接
- **版本历史**：版本历史和变更记录应该单独管理

### 5.3 平衡原则

- **内容完整性**：确保文档内容完整，不遗漏重要信息
- **长度合理性**：保持文档长度合理，不包含冗余信息
- **用户体验**：优先考虑用户的阅读体验和信息获取效率
- **维护成本**：考虑文档的长期维护成本

## 6. 总结

文档长度规范是确保 Skill 文档质量的重要因素。通过遵循适当的长度限制和拆分方法，可以创建更加清晰、易读、易于维护的文档。在实践中，应该根据具体情况灵活应用这些规范，确保文档既完整又简洁，既全面又易于理解。
